# M02 LLM 全平台接入

## 学习目标

1. 理解 OpenAI 兼容、Anthropic Messages、Gemini 三类协议在鉴权、消息结构、流式事件与工具调用上的差异。
2. 设计业务层无感知的 `llm.Provider`：统一同步调用、流式调用、能力声明以及请求/响应数据结构。
3. 实现一套可复用的 OpenAI 风格客户端，并通过配置接入至少一个兼容平台（建议 DeepSeek）。
4. 掌握非兼容协议的适配方式：把 Claude 的请求和响应差异收敛在 Provider 内部。
5. 能为模型调用加入 SSE 解析、Token/成本统计以及主备模型故障转移。

## 练习任务

1. 建立最小 `llm` 包。
    - 定义 `Message`、`ChatRequest`、`ChatResponse`、`StreamChunk`、`Capability` 与 `Provider` 接口。
    - `Provider` 至少包含 `Name`、`Capabilities`、`Chat` 和 `ChatStream`。
    - 让 `Temperature` 使用 `*float64`，分别测试“未设置”和“显式为 0”的序列化结果。
2. 完成 OpenAI 风格 Provider。
    - 基于 M01 的生产级 HTTP 客户端，请求 `/chat/completions`，解析文本与 usage。
    - 将名称、`baseURL`、API Key 收拢到配置结构，避免硬编码。
    - 用 `httptest` 覆盖成功、401/400、429、非法 JSON 与空 `choices` 五种情形。
3. 复用兼容层接入第二个平台。
    - 仅替换 `baseURL`、API Key、模型名，接入 DeepSeek、豆包、Qwen、Kimi 或 GLM 之一。
    - 对比两次请求的统一输入、统一输出和服务商原始错误；记录确实需要单独适配的字段。
4. 编写 Claude 请求适配器。
    - 将统一消息拆成顶层 `system` 与 `user`/`assistant` 消息，补齐必填 `max_tokens`。
    - 添加 `x-api-key` 与 `anthropic-version` 请求头。
    - 对当前尚未表达的 `tool` 消息返回明确错误，不要静默错误转发。
5. 实现并验证流式输出。
    - 编写通用 SSE 读取器：按空行聚合事件、合并多行 `data:`、忽略心跳、提高 Scanner 缓冲区。
    - 分别实现 OpenAI `[DONE]` 和 Claude `message_stop` 的增量解析；异常 EOF 必须通过 `StreamChunk.Err` 暴露。
    - 确认生产者关闭 channel，发送时监听 `ctx.Done()`。
6. 实现 `llmrouter` 命令行小工具。
    - 从环境变量装配可用 Provider，配置主用与备用模型；未配置的 Key 不启用。
    - 先实现按优先级故障转移，输出“实际 Provider、输入/输出 Token、估算成本”。
    - 选做：实现按价格或延迟排序，并记录 P50/P95 延迟。

## 练习思路

1. 先守住抽象边界。
    - 业务层只构造 `ChatRequest` 并调用 `Provider`，不出现供应商名称、鉴权头或协议 JSON。
    - 统一结构只包含当前需要的能力；工具调用、多模态等后续需求不提前塞进接口。
2. 以 OpenAI 兼容协议作为可运行主线。
    - 先让单次 `Chat` 成功，再做 usage、错误解析和测试，最后加入 `ChatStream`。
    - 接入第二家兼容平台时，优先验证“只改配置即可复用”；只有协议扩展确实不同才加小范围补丁。
3. 将异构协议当作翻译问题。
    - Claude 适配器负责把统一请求转换为 Messages API，再把 content blocks 拼回统一响应。
    - 适配器内显式拒绝无法正确映射的数据，避免看似成功、实际语义错误。
4. 分层处理流式数据。
    - 传输层只做 SSE 行协议与事件边界；Provider 层只解析自己关心的 JSON 事件。
    - 为正常结束标记设置断言，网络中断、JSON 解析失败和用户取消要能区分。
5. 把路由当作可治理能力。
    - 工厂按配置生成 Provider；路由策略只决定尝试顺序；Router 负责调用、取消判断与最终错误聚合。
    - 成本以 Provider 返回的 usage 为准，本地估算仅用于预算提醒；降级时同时切换正确的供应商模型名。

## 问题回答

1. 流式请求失败时，当前 Router 还没有实现 ChatStream 的重试和降级。如果模型吐到一半断流，你会整段重来，还是从断点续？这两种方案分别会带来什么问题？

2. Prompt Caching 是前缀匹配。如果每次请求都在 system 最前面插一个变化的时间戳，缓存命中率会怎样？这对提示词顺序有什么启示？

3. “路由 + 模型名”是耦合的：降级到 Claude 时需要使用 Claude 的模型名。你会如何设计，让上层只表达“给我一个便宜的中文模型”，而不关心具体供应商？
